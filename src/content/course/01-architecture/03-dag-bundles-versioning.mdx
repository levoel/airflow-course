# DAG Bundles и Versioning

До 3.0 в Airflow была фундаментальная проблема целостности: **код DAG менялся независимо от runs**. Если вы поменяли `transform()` после того, как DagRun успешно завершился, и потом сделали `clear`, run перезапустится на **новом** коде. Это могло привести к неконсистентности результатов и сбивать backfill в production.

AIP-63 (DAG Versioning) и AIP-66 (DAG Bundles & Parsing) решают эту проблему системно.

---

## DAG Bundles — pluggable backends для DAGs

В 2.x DAGs жили в одной папке `$AIRFLOW_HOME/dags/`. Все DAGs парсились одним DagBag — простая модель, плохая для multi-team setups.

В 3.x **DAG Bundle** — pluggable namespace для DAGs с собственным backend, refresh strategy и versioning:

```yaml
# config.yaml
dag_processor:
  dag_bundles:
    - name: production
      classpath: airflow.dag_processing.bundles.git.GitDagBundle
      kwargs:
        repo_url: 'git@github.com:company/airflow-dags.git'
        ref: main
        refresh_interval: 60

    - name: analytics-team
      classpath: airflow.dag_processing.bundles.s3.S3DagBundle
      kwargs:
        s3_url: 's3://airflow-dags/analytics/'
        refresh_interval: 300

    - name: ml-team
      classpath: airflow.dag_processing.bundles.gcs.GCSDagBundle
      kwargs:
        gcs_url: 'gs://airflow-dags/ml/'
        refresh_interval: 300
```

Каждый bundle:
- Имеет **manifest** (метаданные: id, version, last_modified)
- Содержит **версию** — git commit hash, S3 object version, GCS generation
- **Refresh-ится независимо** от других bundles
- Доступен в UI с фильтром по bundle_id

Bundle backends — pluggable через entry points. Community supports: Git, S3, GCS, Local, HTTP. Можно написать свой (Vault, Artifactory, custom git LFS).

В metadata DB появилась таблица `dag_version`:

```sql
SELECT id, dag_id, bundle_id, version_number, created_at, bundle_url
FROM dag_version
WHERE dag_id = 'orders_etl'
ORDER BY created_at DESC LIMIT 5;
```

DagRun теперь имеет колонку `dag_version_id`. Когда вы `clear` старый run, он запустится на **своей** версии кода — не на текущей.

### Что это меняет на практике

**До 3.0** (сценарий боли):
1. DagRun #1 успешно завершился вчера на коде v1
2. Сегодня вы поменяли transform() — теперь это v2
3. Решили rerun-нуть вчерашний run для исправления данных
4. **Rerun использует v2!** Результат может отличаться от ожидаемого
5. Backfill через 6 месяцев — ещё хуже, код тысячи раз поменялся

**После 3.0**:
1. DagRun #1 на v1
2. `clear` → run на v1 (не на текущей)
3. Можно посмотреть diff между v1 и v2 в UI
4. Backfill сохраняет историческую корректность

## Migration с 2.x → 3.x для DAG storage

Если у вас 2.x с DAGs в локальной папке, в 3.x это становится **Local Bundle**:

```yaml
dag_processor:
  dag_bundles:
    - name: default
      classpath: airflow.dag_processing.bundles.local.LocalDagBundle
      kwargs:
        path: /opt/airflow/dags
```

Но рекомендуется на git/S3 — для production reproducibility, audit, и DAG Versioning (Local Bundle не поддерживает versioning через filesystem ts).

### gitSync sidecar pattern (legacy 2.x) vs Bundles

В 2.x на K8s часто использовали gitSync sidecar: каждый pod (scheduler, worker) имел git pull container. С Bundles это не нужно — DAG Processor сам читает git через `GitDagBundle`. Меньше движущихся частей.

## DAG Processor — внутренности

DAG Processor — отдельный mandatory процесс в 3.x (`airflow dag-processor`):

```python
# Псевдокод main loop
while not should_exit:
    for bundle in dag_bundles:
        if bundle.needs_refresh():     # gitfetch / S3 list
            new_version = bundle.refresh()

            for dag_file in bundle.list_files():
                # Spawn subprocess для parse (изоляция)
                with multiprocessing.Process(parse_file, dag_file) as p:
                    serialized_dag = p.run()

                # Сохраняем в БД
                upsert_serialized_dag(serialized_dag, bundle_version=new_version)
                create_dag_version(dag_id, bundle.id, new_version)

    sleep(parsing_processes_check_interval)
```

Параметры конфига:

| Параметр | Default | Что |
|---|---|---|
| `parsing_processes` | 2 | Parallel workers (рекомендация: 2× vCPU) |
| `min_file_process_interval` | 30 | Не парсить файл чаще |
| `dag_dir_list_interval` | 300 | Refresh bundles периодически |
| `dag_processor_manager_log_location` | ... | Где логи |
| `parsing_pre_import_modules` | True | Pre-import тяжёлых модулей в parent — speedup |

## Multi-Team setup через Bundles (AIP-67)

В Airflow 3.2 (experimental) появилась поддержка **team-scoped bundles** через AIP-67:

```yaml
dag_processor:
  dag_bundles:
    - name: team-a
      team: data-engineering
      classpath: ...
    - name: team-b
      team: ml-platform
      classpath: ...

# Auth через RBAC + team scoping
core:
  multi_team: true
  default_team: data-engineering
```

Каждая команда видит свои DAGs, pools, connections, variables — изоляция в одном Airflow instance. Полезно для enterprise: централизованное администрирование, но dev-teams не пересекаются.
